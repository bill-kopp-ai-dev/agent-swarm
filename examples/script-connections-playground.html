Script Connections MVP

PR #934 · feat/script-connections-mvp · register OpenAPI / GraphQL / MCP / OAuth-backed APIs once, use them from any swarm script via ctx.api / ctx.mcp. This page documents an end-to-end walkthrough of the feature (real calls, real outputs) and includes a playground wired to your local server. Register the connections from the script connections guide to make the presets work.
✓ OpenAPI by spec URL ✓ OAuth credential bindings ✓ ctx.mcp server-side proxy ✓ GraphQL connections

How it fits together

lead agent registers connections (MCP tools, lead-only):
  script-connections   → upsert-openapi | upsert-mcp | upsert-graphql | list | refresh | disable
  credential-bindings  → upsert | oauth-app-upsert | oauth-authorize-url | list | disable
  set-config           → store secrets (encrypted at rest w/ SECRETS_ENCRYPTION_KEY)

any script then gets typed clients, credentials injected at egress only:
  ctx.api.<slug>.<operationId>(args)      // OpenAPI
  ctx.api.<slug>.graphql(query, vars)     // GraphQL
  ctx.mcp.<slug>.<toolName>(args)         // MCP (proxied through the API server)

Step 1 — OpenAPI connection + credential binding

Registered the swarm's own REST API against itself: spec fetched from /openapi.json (117 routes → generated operations), secret stored encrypted, bound as a header.

// 1. store the secret (encrypted at rest)
set-config { "key": "SWARM_SELF_KEY", "value": "123123", "isSecret": true, "scope": "global" }

// 2. register the connection + binding in one call
script-connections {
  "action": "upsert-openapi",
  "slug": "swarm",
  "baseUrl": "http://localhost:3013",
  "openapiSpecUrl": "http://localhost:3013/openapi.json",
  "configKey": "SWARM_SELF_KEY",
  "headerTemplate": "Authorization: Bearer [REDACTED:SWARM_SELF_KEY]",
  "allowedHosts": ["localhost"]
}
export default async function(args, ctx) {
  const res = await ctx.api.swarm.getApiAgents();
  return { agentCount: res.agents.length, names: res.agents.map(a => a.name) };
}
{ "agentCount": 1, "names": ["demo-lead"] }
Gotcha found live: the header template must contain the literal placeholder [REDACTED:KEY]. I wrote {{SWARM_SELF_KEY}} first → the binding is silently skipped → 401. Tool description doesn't show the format. (Also registered Petstore by spec URL — worked, but petstore3.swagger.io itself was returning 500s that day.)

Step 2 — MCP connection (ctx.mcp)

Registered the swarm's own MCP endpoint as a server, then a connection with tool discovery: 117 tools found, camelCased into methods. Calls are proxied server-side — the MCP server's auth headers never reach the script subprocess.

mcp-server-create { "name": "swarm-self-mcp", "transport": "http", "url": "http://localhost:3013/mcp", "headers": "{…auth…}", "scope": "swarm" }
script-connections { "action": "upsert-mcp", "slug": "swarmmcp", "mcpServerId": "5ae4fe03-…" }
const metrics = await ctx.mcp.swarmmcp.getMetrics({});
{ "content": [{ "type": "text", "text": "Swarm metrics: 1 tasks, 1 agents, …" }],
  "structuredContent": { "tasks": { "total": 1 }, "agents": { "total": 1 }, "skills": { "total": 9 }, … } }
Known MVP gap: you get the raw MCP envelope (content + structuredContent) rather than an unwrapped result.

Step 3 — GraphQL connection

script-connections { "action": "upsert-graphql", "slug": "countries", "baseUrl": "https://countries.trevorblades.com/", "allowedHosts": ["countries.trevorblades.com"] }
const data = await ctx.api.countries.graphql(
  "query($code: ID!) { country(code: $code) { name capital currency emoji languages { name } } }",
  { code: "ES" }
);
{ "country": { "name": "Spain", "capital": "Madrid", "currency": "EUR", "emoji": "🇪🇸",
  "languages": ["Spanish", "Basque", "Catalan", "Galician", "Occitan"] } }
Signature is positional — graphql(query, variables), not an options object. And allowedHosts is required even with no credential attached.

Step 4 — OAuth flow, end to end

Done against a local fake provider on :4457 (identical code path to GitHub/Slack/etc — only the URLs differ).

// 1. register the OAuth app (client id/secret from the provider's dev console)
credential-bindings { "action": "oauth-app-upsert", "provider": "fakehub", "clientId": "demo-client-id", "clientSecret": "…",
  "authorizeUrl": "http://localhost:4457/authorize", "tokenUrl": "http://localhost:4457/token", "scopes": ["read"] }

// 2. get the URL to open in a browser — state + PKCE S256 generated by the swarm:
credential-bindings { "action": "oauth-authorize-url", "provider": "fakehub" }
→ http://localhost:4457/authorize?client_id=…&redirect_uri=…/api/oauth/fakehub/callback
    &response_type=code&scope=read&state=cEUyFV…&code_challenge=nV2lUX…&code_challenge_method=S256

// 3. user authorizes → provider redirects to the swarm callback → code exchanged
//    (client secret + PKCE verifier) → access+refresh token stored → "you can close this tab"

// 4. bind the token and attach to a connection
credential-bindings { "action": "upsert", "configKey": "FAKEHUB_TOKEN", "authKind": "oauth", "oauthProvider": "fakehub",
  "allowedHosts": ["localhost"], "headerTemplate": "Authorization: Bearer [REDACTED:FAKEHUB_TOKEN]" }
script-connections { "action": "upsert-openapi", "slug": "fakehub", "baseUrl": "http://localhost:4457", …, "credentialBindingId": "2d57ec26-…" }
const me = await ctx.api.fakehub.whoami();
{ "login": "taras-demo", "plan": "pro", "authedVia": "oauth" }

Security proof

The provider's access log shows the real token arriving: Bearer fakehub-access-token-e2e-1a2b3c4d.
An exfiltration probe sending the same placeholder to httpbin.org (not allow-listed) arrived as the literal string Bearer [REDACTED:FAKEHUB_TOKEN]. Substitution happens only at fetch egress, only for allow-listed hosts — the script process never holds the secret, and tokens echoed to stdout are scrubbed.
